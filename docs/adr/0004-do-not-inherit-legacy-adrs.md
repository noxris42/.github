# 0004. 旧ADRをLegacyとして継承しない

## Metadata（メタデータ）

| 項目 | 内容 |
| --- | --- |
| Status | Accepted |
| Date | 2026-07-14 |

---

## Context（背景）

このプロジェクト（個人アカウント用`.github`リポジトリ）の設計を進める過程で、以前に作成していた別のリポジトリの12件のADRをどう扱うかが論点になった。

当初、`docs/adr/legacy/github-personal/`のようなフォルダを設けて旧ADRを取り込み、分類する案が検討され、実際に一部の分類作業まで進んだ。あわせて、旧ADR・現行Documentation・設計チャットから新しいADRを「採掘」するための中間工程（ADR Mining）を設ける案も提示された。

しかし、この案を採用した場合の影響を見直したところ、次の懸念が見えてきた。このプロジェクト単体で見ればLegacyフォルダは自然に見えるが、一歩引いて見ると、このプロジェクトだけが特別扱いされていることになる。将来、他のプロジェクトも版を重ねていけば（例：あるプロジェクトのv1からv2への移行）、それぞれが同様のLegacyフォルダを持つことになりかねない。これは知識の系譜を管理する仕組みというより、単なる履歴管理の乱立である。

このリポジトリは、旧リポジトリを改造・拡張したものではなく、設計の考え方を一度リセットしてゼロベースで再設計されたものである。旧リポジトリの知識が今回の設計を考える上で影響した場面はあったとしても、それは通常の経験・知識が新しい設計に影響するのと同じことであり、正式なADRの親子関係として管理する必要はない。

---

## Decision（決定）

このプロジェクトのADRは、旧リポジトリのADRをLegacyとして継承せず、このプロジェクト内で確定した設計判断のみを記録する。

過去のADRを参考にすること自体は妨げないが、正式な派生関係・継承関係としてこのプロジェクトのADR体系へ組み込まない。各プロジェクトは、それぞれ自身のDecision Historyを持つ。

---

## Rationale（理由）

過去との関係性・知識の系譜を横断的に管理する機能を、個々のプロジェクトのADR体系ごとに持たせると、プロジェクトが版を重ねる・増えるたびに同種の管理構造が乱立する。これは局所最適（このプロジェクト単体では整って見える）ではあっても、全体最適（複数プロジェクトを横断した知識管理）にはならない。

過去との関係性・知識の系譜を横断的に扱う機能は、個々のプロジェクトのADR体系の責務ではなく、そのための別の仕組みの責務とすべきである。このプロジェクトのADRは、このリポジトリ自身の設計判断履歴だけを扱うことで、自己完結性を保つ。

---

## Alternatives Considered（却下した案）

| 選択肢 | 採用しなかった理由 |
| --- | --- |
| `docs/adr/legacy/`で旧ADRを保存する | このプロジェクト単体では合理的に見えるが、将来他のプロジェクトも同様の構造を持つことになれば、履歴管理が乱立する。局所最適に留まり、全体最適にならない |
| 旧ADRを再番号付けして現行ADR体系へ取り込む | このプロジェクトがゼロベースで再設計されたという前提と矛盾する。継承関係を持たせると、現行ADRが旧ADRを読まなければ理解できなくなり、自己完結性が損なわれる |
| ADR Miningという中間工程を設け、旧ADR・現行Documentation・チャットから新ADRを抽出する専用フォルダを置く | Legacy ADRそのものを持たない方針としたため、この中間工程も不要になった |

---

## Consequences（影響・注意点）

- 旧リポジトリの12件のADRは、このリポジトリには含まれない
- 過去の設計判断から得た知見は、新しいADRのContext・Rationaleに、このプロジェクト自身の言葉で直接記述してよい。旧ADRファイルへの参照リンクは張らない
- 過去との関係性・知識の系譜の管理は、このプロジェクトのADR体系の責務外とする
