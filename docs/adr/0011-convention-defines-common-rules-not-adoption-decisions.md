# 0011. Conventionは採用後の共通ルールのみを定義し、採用の可否は扱わない

## Metadata（メタデータ）

| 項目 | 内容 |
| --- | --- |
| Status | Accepted |
| Date | 2026-07-14 |

---

## Context（背景）

Branch運用の設計を検討する過程で、既存の参考資料にあったブランチモデルが、リポジトリの用途によって適否が大きく異なることが分かった。ドキュメント管理用途では`main`のみの運用が適切である一方、Webアプリケーションでは`main`と`hotfix/*`の組み合わせが適切、というように、採用すべき運用モデル自体がリポジトリごとに異なっていた。

この違いに対処しようとして、当初はConvention文書の中でモデルごとに詳細な規約まで書き分ける案が検討された。しかしこれを進めると、コミット規約もワークフローもモデルごとに分岐させる必要が生じることに気づき、責務の整理が必要になった。

検討の結果、「Branchを使うならこの命名基準に従う」「Branchに Issue番号を含むならこの順番にする」という共通ルールと、「どの運用モデルを採用するか」「LTS Release Branch Flowのような具体的なフローをいつ使うか」という採用判断は、別の階層の話であることが整理された。

---

## Decision（決定）

Conventionは、ある概念・仕組みを採用した場合に共通して守るべきルールのみを定義する。その概念・仕組みを採用するかどうか、どの構成・フローで運用するかの判断は、Convention文書の対象外とする。

例えば、Branch Conventionが存在することは、すべてのリポジトリでBranch運用を必須にすることを意味しない。Branchを使うと決めたリポジトリが、そのブランチをどう命名し、どう扱うかを定義するのがBranch Conventionの責務である。

どの仕組みを採用し、どのように構成・適用するかは、Repository Setupや各Specificationの責務とする。Templateはそのために利用される構成部品を提供する。

---

## Rationale（理由）

「採用するかどうか」と「採用した場合にどう扱うか」を同じ文書に混在させると、リポジトリの用途ごとに条件分岐した規約を書く必要が生じ、Convention文書が用途別の分岐だらけになる。これはConventionを「共通認識・共通ルール」として機能させる上で不都合であり、規模が大きくなるほど保守が難しくなる。

Conventionを「採用後のルール」に限定することで、Convention自体はリポジトリの用途によらず一定の内容を保てる。採用するかどうかの判断は、リポジトリ構築時の設計判断（Repository Setup）や、具体的な構成部品（Template）の選択に委ねることで、責務が分離される。

---

## Alternatives Considered（却下した案）

| 選択肢 | 採用しなかった理由 |
| --- | --- |
| Convention文書内で、運用モデルごとに詳細な規約まで書き分ける | モデルごとの分岐がConvention文書内に持ち込まれ、関連するコミット規約・ワークフローも同様にモデルごとの分岐が必要になる。Conventionが用途別の分岐だらけになり、共通ルールとしての一貫性が保てない |

---

## Consequences（影響・注意点）

- Convention文書に「このリポジトリでは何を採用すべきか」という判断は書かない。Conventionが定義するのは、採用した場合に従うべきルールのみである
- 「どの仕組みを採用するか」の判断は、Repository Setup・Specificationの責務とする。Templateは採用された仕組みを実装する際に利用される構成部品を提供する
- 新しい概念・仕組み（新しいBranch運用モデル等）が必要になった場合、まずその概念を採用するかどうかの判断（Repository Setup側）と、採用した場合の共通ルール（Convention側）を分けて設計する
