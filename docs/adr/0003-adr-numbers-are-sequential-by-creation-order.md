# 0003. ADR番号は作成順の単純連番とする

## Metadata（メタデータ）

| 項目 | 内容 |
| --- | --- |
| Status | Accepted |
| Date | 2026-07-14 |

---

## Context（背景）

このプロジェクトの初期には、ADR番号を「論理構造順」で採番する方針が採られていた。これは、番号の並びによってADR同士の構造的な関係を表現しようとする考え方であり、命名規則側（ファイル名の採番形式）とは別に、番号が持つ意味そのものをADRの設計思想として定義していた。

この方式には、新しい決定を追加するたびに、既存の番号体系の中で意味的に正しい位置を検討する必要があるという負担があった。ADRを増やすこと自体への心理的な負荷が高まり、「独立して覆せる決定が確定したら、その都度気軽にADRを追加する」という運用と相性が悪かった。

---

## Decision（決定）

ADR番号は、確定した設計判断をADRとして記録した順の単純連番とする。番号は記録を一意に識別するための識別子に徹し、判断の重要度・依存関係・論理構造を表現しない。

具体的な採番形式（ゼロパディング桁数、ファイル名パターン、欠番の扱い等）は本ADRの対象外とし、`docs/adr/README.md`が定義する。

---

## Rationale（理由）

番号に構造的な意味を持たせると、新しい決定を追加するたびに既存の番号体系との整合を検討する必要が生じる。これはADRを増やすことそのものへの負荷となり、「独立して覆せる決定が確定した時点で記録する」というADRの作成タイミングの原則（`docs/philosophy/adr.md`参照）と相性が悪い。

番号を単純な作成順の識別子に限定すれば、番号の意味を考えずにADRを追加できる。決定同士の関連は、番号の位置ではなく、本文（Context・Related Documents等）の相互参照によって表現する。

---

## Alternatives Considered（却下した案）

| 選択肢 | 採用しなかった理由 |
| --- | --- |
| 論理構造順で採番する（過去に採用していた方式） | 新しい決定を追加するたびに、既存の番号体系の中で意味的に正しい位置を検討する必要が生じる。ADRを追加すること自体への心理的負荷が高まり、番号の整合性を保つための調整コストが発生する |
| カテゴリ・領域ごとに番号帯を分ける | 論理構造順と同様に、新しい決定がどの番号帯に属するかという判断を都度要求する。決定の対象領域は後から変わることもあり、番号帯による分類はその変化に追従しにくい |

---

## Consequences（影響・注意点）

- ADR番号から、判断の重要度・依存関係・論理構造を読み取ることはできない。決定同士の関連は、本文の相互参照によって確認する
- ADR番号を、後から意味のある順序に並べ替えたり、体系を整えたりする作業は行わない
- 番号の具体的な運用細則（ゼロパディング、ファイル名規則、欠番の扱い等）は`docs/adr/README.md`が引き続き所有する
