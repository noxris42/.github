# 0010. `templates/`を再利用可能な部品として管理する

## Metadata（メタデータ）

| 項目 | 内容 |
| --- | --- |
| Status | Accepted |
| Date | 2026-07-14 |

---

## Context（背景）

`templates/`領域をどう構成するかを検討する過程で、複数の案が試された。

当初は、リポジトリの用途ごとに完成済みのテンプレート一式を用意し、リポジトリ構築時にそのままコピーする案が想定されていた。

次に、`base / flow / audience / visibility / management`のような複数の分類軸を組み合わせてテンプレートを構成する案が検討された。しかし`visibility`・`management`軸の具体的な使い道が定まらず、抽象的な分類だけでは実際の運用場面を想定できないまま行き詰まった。

この行き詰まりを受けて、実際に構築されうる具体的なリポジトリ例（個人事業管理・非公開・日本語等）を先に書き出し、そこから共通点・差分を分析するボトムアップの検討に切り替えた。この分析を経て、テンプレートファイルをリポジトリごとに複製する方式ではなく、リポジトリ構築の過程で必要な部品を取得し、組み合わせて使う方式へ転換した。

---

## Decision（決定）

`templates/`が管理するテンプレート資産を、利用時に組み合わせ・調整できる再利用可能な部品として管理する。

完成済みのテンプレート一式をリポジトリ用途ごとに用意するのではなく、成果物の種類（Workflow・Action・Label・Issue Template・Pull Request Template・Document等）ごとに部品を管理し、リポジトリ構築時に必要な部品を選択して取得・編集・利用する。

---

## Rationale（理由）

用途ごとの完成済みテンプレート一式をコピーする方式では、`.github`リポジトリ内に用途別の完成セットを大量に重複して保持することになる。組み合わせのパターンが増えるたびに、同一の成果物を複製したテンプレートセットを新たに維持する必要が生じる。

テンプレートを部品として管理することで、`.github`リポジトリ内に用途ごとの完成済みセットを重複して保持する必要がなくなる。共通する成果物は単一の部品として再利用し、リポジトリごとの差分は利用時の選択・調整として扱える。また、部品を成果物の種類で分類することで、将来新しい成果物の種類が必要になった場合にも、既存の部品体系を崩さずに追加できる。

---

## Alternatives Considered（却下した案）

| 選択肢 | 採用しなかった理由 |
| --- | --- |
| リポジトリ用途ごとに完成済みテンプレート一式を用意し、コピーする | `.github`リポジトリ内に用途別の完成セットを大量に重複して保持することになり、組み合わせのパターンが増えるたびに同一成果物を複製したセットを新たに維持する必要が生じる |
| `base / flow / audience / visibility / management`という複数軸でテンプレートを分類・組み立てる | `visibility`・`management`軸の具体的な使い道が定まらず、抽象的な分類軸だけでは実際の運用場面を想定できなかった。具体的なリポジトリ例からのボトムアップ分析を経て、この多軸分類自体を採用しない判断に至った |

---

## Consequences（影響・注意点）

- `templates/`は成果物の種類によってディレクトリを構成し、部品ごとに独立して意味を持つ構造とする
- Template部品はConvention本文を複製しない。規約の正本はConventionが持つ
- 部品の具体的なディレクトリ構造、取得方法（コピー・参照・生成）、リポジトリへの適用手順は、本ADRの対象外とし、`docs/architecture/template.md`・`docs/specifications/repository-setup.md`が定義する
