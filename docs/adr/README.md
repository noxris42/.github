# ADR（アーキテクチャ決定記録）

本ディレクトリは、確定した設計判断について、その背景・理由・却下した案・影響を記録するADR（Architecture Decision Record）を管理する。

ADRの存在理由・価値観は `docs/philosophy/adr.md` を参照する。本書はADRの具体的な運用方法（何を記録し、どう分割し、どう状態管理し、どう採番するか）を定義する。

---

## ADRの記録対象（記録要否）

将来、別の選択肢を採用しなかった理由を説明する必要がある設計判断をADRの記録対象とする。

判断に迷う場合は、次の問いに置き換えて考えるとよい。

> 数か月後に異なる案を提案されたとき、「なぜ現在の案を選んだのか」を説明する必要があるか。

Yesであれば記録対象の候補になる。

### ADR化しないもの

以下はADRの記録対象としない。design-notesに留めるか、あるいは記録自体が不要である。

- 見出しの細かな文言・誤字修正
- 既存Conventionに従っただけの選択
- 実装上自明なファイル配置
- 既存ADRから必然的に導かれる詳細
- 可逆性が高く、設計上重要でない判断

「記録対象になるかどうか」と「記録対象を何本に分けるか」は別の基準である。後者は次節（ADRの単位）を参照する。

## ADRの単位（粒度判定）

議題単位ではなく、決定単位で1ファイルとする。以下の基準で判定する。

### 主基準（いずれかに該当したら分割する）

**Status単一性テスト**
そのファイル内容全体に対して、単一のStatus（Proposed/Accepted/Superseded/Deprecated）で正直に答えられるか。答えられない場合、複数の決定が混在している。

**独立可逆性テスト**
一部の決定だけを変更した場合でも、残りの決定をそのまま有効として維持できるか。維持できるなら、それは別の決定として分割する。判断基準は「関連箇所の確認が必要か」ではなく、論理的に同時変更が必要かどうかである。

### 補助基準

**参照単一性テスト**
他の文書から、その決定だけを独立して参照したくなる可能性があるか。あるなら別ADRである方が自然である。ただし主基準2つで判断がつく場合は、この基準を持ち出さない。

### 1つのADRに含めてよいもの／分割すべきもの

| 1つのADRに含めてよい | 分割すべき |
| --- | --- |
| 決定を成立させる詳細条件・適用範囲・例外条件 | 別々の理由で選択されたもの |
| 決定によって必然的に導かれる構造 | 片方だけ変更しても他方が成立するもの |
| 同じStatusと同じ置換理由を共有する決定群 | 別の時期に廃止される可能性があるもの／別文書から個別参照されるもの／異なる代替案比較から導かれたもの |

### 例外の扱い

例外規則を先に詳細化しない。「単一Statusで管理でき、同時に覆される決定を1ADRとする」という原則だけでまず運用し、実例が蓄積してから必要に応じて補足する。

## ADRを書くタイミング

ADRの記録対象となる判断が確定するタイミングと、ADRを実際に作成する作業タイミングは分けて扱う。

- 記録対象になるタイミング：上記の記録要否基準を満たす設計判断が確定した時点
- 実際に書くタイミング：その決定を前提とした後続作業へ進む前、または設計作業の区切り

決定が出るたびにリアルタイムで中断して書く運用は求めない。1回の議論から複数のADR対象となる独立した決定が確定すれば複数本、複数回の議論を経て1つの決定に収束すれば1本作成する。

`Proposed` として作成する場合は、記録対象となる可能性が高い判断を正式レビューに付すための例外であり、通常の作成タイミングには含めない。

---

## ADRに書く内容／書かない内容

ADRは決定時点の判断・理由・適用範囲・既知の影響を凍結する記録である。未来の情報を完全に排除するわけではなく、現在の決定に含まれる将来条件は書いてよい。

| 書く | 書かない |
| --- | --- |
| 決定の適用条件・適用範囲 | 次に検討したいアイデア |
| 既知の制約 | まだ結論のない候補 |
| 再検討を必要とするトリガー（例：外部コントリビューター受入れ開始時に再検討する） | 現在の運用状況の実況（例：現在は無効） |
| 決定時点で認識していた将来的影響 | 一時的な有効・無効状態、作業予定・TODO |

Rationaleには最終的な採用理由だけでなく、Alternatives Consideredと連携して「なぜ有力だった別案を採用しなかったか」が読める状態を維持する。適用範囲・制約・再検討トリガーを詰め込みすぎないよう、固定の小見出しは設けず、自然な文章の中で識別しやすく書けば十分である。

## ファイル命名規則（命名規則）

```text
<4桁連番>-<kebab-case-slug>.md
```

**Examples:**

```text
0000-template.md
0001-adopt-layer-dependency-direction.md
0002-separate-setup-and-operation.md
```

- 連番は**実際に確定・作成した順**に4桁のゼロパディングで採番する
- 番号はカテゴリ順・論理構造順に整えようとしない。番号は識別子であり意味を持たない（`docs/philosophy/adr.md` 参照）
- スラッグは判断の主題を英語のkebab-caseで表現する
- ファイル名に `adr-` プレフィックスは付けない（このディレクトリ配下である時点でADRと判断できるため）

## Statusの定義（ステータス定義）

| Status | 意味 |
| --- | --- |
| `Proposed` | 提案済みだが未確定。正式レビューを経て確定する運用を取る場合にのみ使用する一時的な状態 |
| `Accepted` | 採用された。原則として、決定確定後にこのStatusで作成する |
| `Superseded` | 後続ADRにより置換された。`Superseded by` に置換先のADR番号を併記する |
| `Deprecated` | 置換先のADRはないが、現在は推奨・適用されない |

ADRは原則として決定確定後に `Accepted` として作成する。`Proposed` は例外的な状態であり、正式レビューを経て確定させる運用を取る場合にのみ使用する。

`Deprecated` と `Superseded` の違い：後継の決定が明示的に存在する場合は `Superseded`、後継なしにその決定自体が不要になった場合は `Deprecated`。

`Rejected` というStatusは設けない。採用しなかった案はAlternatives Consideredに残す。却下された提案そのものを長期保存する必要が生じた場合にのみ、将来追加を検討する。

## ADR間の関係表現

正式に表現するのは置換関係のみとする。`Status: Superseded` と `Superseded by: 000X` をMetadataに記載する。

前提・派生・依存・関連は、本文（Context・Consequences）で自然文として参照すれば足りる。専用の必須セクションにはしない。

`Partially Superseded` のような新しいStatus値は追加しない（Status単一性の原則を崩すため）。一部の決定だけが差し替わる場合は、その部分を独立した決定として最初から分割しておくことで対処する。

## テンプレートの使い方（テンプレート利用方法）

`0000-template.md` をコピーして新しいADRを作成する。

```text
0000-template.md → 0001-<slug>.md
```

セクション構成はテンプレートファイルを参照する。

## 他文書との関係（依存方向）

現行Documentation（Philosophy・Architecture・Convention・Specification）は、内容を成立させるためにADRへ依存してはならない。ただし、判断履歴への補助的な参照は妨げない（例：「判断理由はADR 0003を参照」）。

design-notesとの関係は `docs/design-notes/README.md` および `docs/philosophy/adr.md` を参照する。
