# 0001. 設計検討記録と正式な判断履歴を分離する

## Metadata（メタデータ）

| 項目 | 内容 |
| --- | --- |
| Status | Accepted |
| Date | 2026-07-14 |

## Context（背景）

このプロジェクトでは、確定した設計判断の理由を将来へ説明可能にする正式な記録（ADR）を運用してきた。運用初期にADRを複数作成する過程で、ADRの対象範囲について次の緊張関係が繰り返し生じた。

- ADRを「確定した決定だけを記録するもの」に絞り込むほど、決定に至るまでの比較・議論・未採用案・保留事項が、ADRにもDocumentationにも残らず失われる懸念が生じる
- 逆に、検討過程・知見（Decision Evolution、Design Insightsなど）までADRに含めようとすると、ADRが確定判断の記録ではなく議事録に近づき、「現在有効な規範ではない正式な履歴記録」というADRの立ち位置が曖昧になる

この緊張関係への対処として、過去には「Design Record」という中間資産（設計検討をADRとは別に保存する仕組み）が導入されたことがあった。しかしDesign Recordは、テンプレート・Front Matter・Review Historyなど正式な知識体系として厳密に設計され、結果としてADRに近い重みを持つ資産になっていった。この重厚化には、「あらゆる知識・経験を失わずデータ化して蓄積したい」という、このリポジトリの外側にある別の構想（知識管理基盤）由来の価値観が影響していた。

この状態を踏まえ、設計検討の保存先と正式な判断履歴の保存先を、責務と成熟度の異なる別資産として改めて設計し直す必要があった。

---

## Decision（決定）

正式な判断履歴と非規範的な設計検討記録を、異なる責務を持つ別の資産として管理する。

- **ADR**：確定した設計判断について、その理由を将来へ説明可能にする正式な履歴記録
- **design-notes**：確定前の議論・疑問・比較・保留事項を、任意の完成度・形式で保存する非規範的な作業記録

両者は依存関係ではなく、必要に応じた補足参照の関係で結びつける。design-notesは正式な設計判断の根拠として依存されない。ADRはdesign-notesを検討過程の補足資料として参照してよいが、確定した判断とその理由はADR本文だけで理解できる状態（自己完結）を保つ。

design-notesの具体的な命名規則・テンプレート・運用細則は、本ADRの対象外とする。それらは`docs/design-notes/README.md`・`docs/architecture/documentation.md`が定義する現行ルールに委ねる。

---

## Rationale（理由）

設計検討と正式な判断履歴を分離することで、以下が同時に成立する。

- ADRは「確定した判断とその理由」だけに集中でき、Decision Recordとして純化できる
- 検討過程・比較・未採用案・保留事項を残す場所が別に確保されるため、ADRを純化しても情報が失われない
- design-notesはADRへ収束することを要求されないため、結論に至らなかった検討や、後に別の判断へ置き換わった検討も、無理に整形せずそのまま残せる

依存関係ではなく補足参照の関係にした理由は、design-notesが非規範的な資産である以上、ADRの正しさやDocumentationの正しさが、design-notesを読まなければ成立しない状態を避けるため。ADR・Documentationはそれ単体で読んで理解できる状態を維持する。

## Alternatives Considered（却下した案）

| 選択肢 | 採用しなかった理由 |
| --- | --- |
| 設計検討もすべてADRに含める（Knowledge Record的なADR） | ADRが議事録に近づき、「現在有効な規範ではない正式な履歴記録」という位置づけが曖昧になる。ADRの粒度・Statusの一貫性も損なわれる |
| 設計検討を保存する資産を作らない（確定した決定だけを残す） | 決定に至らなかった検討・保留事項が完全に失われ、後から「なぜその案を採用しなかったか」を説明できなくなる |
| 旧Design Record方式（テンプレート・Front Matter・Review History等を備えた正式資産として設計検討を保存する） | 設計検討の保存先自体が正式な知識体系として厳密化され、ADRと同格の重みを持つ資産になった。結果としてADR・Design Record双方の運用コストが上がり、責務の違いが形式の違いに埋没した |
| 「Design Journal（設計検討）→Decision Mining→Decision Record（ADR）」という二段階生成フロー | 設計検討をADR生成の前段階として捉えやすくなり、ADRへ収束しない独立した検討記録としての責務と緊張関係が生じるため |

---

## Consequences（影響・注意点）

- ADRは、設計検討そのもの（Decision Evolution・Design Insightsに相当する内容）を持たない。確定した判断・理由・却下した案・影響のみを記録する
- design-notesは、正式な知識体系としての完成度・一貫性を求められない。内容の不完全・矛盾・陳腐化を許容する
- design-notesに含まれる検討のうち、将来別の選択肢を採用しなかった理由を説明する必要がある設計判断は、ADRへ抽出する。それ以外の内容は、必要に応じて責務を持つ現行Documentation（Philosophy・Architecture・Convention・Specification）へ直接反映してもよく、必ずしもADRを経由しない
- 過去に採用していたDesign Record方式・二段階生成フローは、本ADRの決定により置き換えられた。今後これらの方式へ戻すことは、本ADRを再検討する対象となる
