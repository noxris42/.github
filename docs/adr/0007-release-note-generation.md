# 0007. リリースノート生成方針

日付：2025-04-20
状態：採用

---

## 背景

リリースノートをどのように生成・カテゴリ分けするかの方針を決める必要があった。GitHubの組み込み自動生成機能（`release.yml` + PRラベル）を使う方法と、カスタムワークフローでPRタイトルを直接読み取る方法の2案が候補となった。

また、PRラベルを使う場合、既存の `type:` ラベル（IssueテンプレートとProjectsの自動追加用）との概念的な衝突が問題として浮上した。

## 決定

**PRタイトルから `type` を直接抽出するカスタムワークフロー（`release-draft.yml`）を採用する。**

- トリガーは手動（`workflow_dispatch`）のみとする
- 入力パラメータ：`version`（リリースバージョン）・`previous_tag`（前回タグ、空欄で自動検出）
- 前回タグ以降にmainへマージされたPRを取得し、PRタイトルの `type` でカテゴリ分けしてドラフトリリースを作成する
- `BREAKING CHANGE` の検出はPRテンプレートのチェックボックス（`- [x] あり`）で行う

## 理由

**PRラベル方式（GitHub組み込み + `release.yml`）を不採用にした理由：**

`release.yml` はPRに付与されたラベルを基準にリリースノートをカテゴリ分けする。これを機能させるためには、PRにコミット規約の `type`（`feat` / `fix` 等）に対応したラベルを付与する必要がある。

しかし既存の `type:` ラベルはIssueテンプレートの種別判別（`type:epic` / `type:feature` 等）とProjectsの `Auto-add to project` フィルター用に定義されており、コミット規約の `type` とは概念が異なる。

- `type:feature`（Issue）= Featureテンプレートで作られたIssue
- `feat`（コミット）= 新機能を追加するコミット種別

同じ名前空間を共用すると意味が混在し、将来的に混乱の原因になる。別プレフィックス（`change:` 等）を新設すれば解決できるが、ラベル体系が複雑化する。

**PRタイトル直接抽出方式（案C）を採用した理由：**

PRタイトルは `<gitmoji> <type>(<scope>): <subject>` 形式に統一されており、`ci-pr.yml` のCIでフォーマットを検証している。この情報を直接信頼の源泉にすることで、ラベル体系を一切汚さずにリリースノートを生成できる。PRラベルを介した間接的な仕組みよりも動作が予測しやすい。

**手動トリガーを採用した理由：**

タグpushを自動トリガーにすると、バージョンファイルの更新やタグ作成のタイミングによってワークフローの実行順序が複雑になる。リリースは手動でコントロールする手順（`release-draft.yml` 実行 → ドラフト確認 → Publish）の方が安全であり、事故が防ぎやすい。

## 却下した案

**GitHub組み込み自動生成（`release.yml` + PRラベル）**：PRラベルとIssueラベルの概念衝突が解消できないため却下。別プレフィックスの新設も構造が複雑になるため却下した。

**タグpush自動トリガー**：リリースタイミングの制御が困難になるため却下。

**`CHANGELOG.md` への自動コミット**：ドラフトリリースで確認・編集してからPublishする手順と二重管理になるため不採用。

## 影響・注意点

- PRタイトルが規約フォーマットに従っていない場合、「その他」に収容される。`ci-pr.yml` でフォーマットを事前に検証しているためズレは発生しにくいが、フォーマット外のPRが混入した場合は手動でリリースノートを修正する
- `BREAKING CHANGE` の検出はPRテンプレートの `- [x] あり` チェックボックスのみで行う。コミットメッセージの `BREAKING CHANGE:` フッターはリリースノート生成には使用しない
- ワークフローファイル名は当初 `release-notes.yml` で検討されたが、「リリースノートを生成する」という印象を与えるため `release-draft.yml` に変更した（ドラフトReleaseの作成まで行うことを明示するため）
