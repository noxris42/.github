# 0001. 規約ドキュメントの構成

日付：2025-04-14
状態：採用

---

## 背景

GitHub運用ルールを文書化するにあたり、ドキュメントの分割粒度と配置場所を決める必要があった。細かく分けすぎると参照が増えて使いづらく、まとめすぎると巨大なファイルになり保守しにくい。また、独自ドキュメントを `.github/` 内に置くか、別フォルダに置くかも検討対象だった。

## 決定

以下の4ファイル構成を採用する。

| ファイル | 配置 | 役割 |
| --- | --- | --- |
| `CONTRIBUTING.md` | `.github/` | 入口・ナビゲーション |
| `COMMIT_CONVENTION.md` | `docs/` | コミット規約 |
| `WORKFLOW_CONVENTION.md` | `docs/` | ワークフロー規約（ブランチ・Issue・PR・リリース） |
| `PROJECT_CONVENTION.md` | `docs/` | プロジェクト規約 |
| `REPOSITORY_CONVENTION.md` | `docs/` | リポジトリ規約 |

## 理由

**4ファイルに分割**：参照する場面が異なる情報を分離することで、必要なファイルだけ開けばよい構成にした。ブランチ・Issue・PR・リリースは一連のフローとして同じ場面で参照するため `WORKFLOW_CONVENTION.md` に統合した。

**独自ドキュメントを `docs/` に配置**：`.github/` はGitHubが自動参照する特殊ファイルを置く場所であり、その意味を壊さないよう独自ファイルは `docs/` に分離した。`CONTRIBUTING.md` はGitHubが自動認識してIssue/PR作成時にリンクを表示する特殊ファイルのため `.github/` に残し、実質的な規約は `docs/` 内のファイルへの目次として機能させた。

**`docs/` を選択**：`conventions/` はフォルダ名が規約限定で拡張性が低い。`docs/` は将来ADRや設計書などを追加しやすく、GitHubも認識するフォルダ名として一般的。

## 却下した案

**`CONTRIBUTING.md` にすべてまとめる案**：ファイルが長くなりすぎ、コミット規約だけを参照したい場面で目的のセクションを探す手間がある。

**7ファイル細分化案**（BRANCH / ISSUE / PR / RELEASE を個別化）：参照が増えすぎて日常的な確認に不向き。

**独自ドキュメントを `.github/` 内に配置**：`.github/` の本来の意味（GitHubが自動認識する場所）が曖昧になり、新参者が混乱しやすい。

**`docs/conventions/` にネストする案**：サブフォルダを1段挟む分だけパスが深くなる。現時点での規約ファイル数では `docs/` 直下で十分。

## 影響・注意点

- ブランチ命名規則は `WORKFLOW_CONVENTION.md` を定義元とし、`REPOSITORY_CONVENTION.md` は参照のみとする（重複定義による不整合を防ぐ）
- ドキュメント間のクロスリファレンスが発生するため、リンク切れに注意する
