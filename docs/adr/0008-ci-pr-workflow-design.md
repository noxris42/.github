# 0008. `ci-pr.yml` の検証対象とブランチ名ルールの設計

日付：2025-04-20
状態：採用

---

## 背景

PRマージ前に規約準拠を自動検証するCIワークフローを設計するにあたり、以下の論点を整理する必要があった。

- 何を検証対象とするか（PRタイトル・ブランチ名・コミットメッセージ・ファイルパスなど）
- ブランチ名の検証ルールをどのように設計するか（typeごとのパターン・例外の扱い）
- `⚡️` の複数バイト絵文字をどう扱うか

## 決定

**ワークフロー構成：**

```
ci-pr.yml
  ├── job: validate-pr-title    # PRタイトルの検証
  └── job: validate-branch-name # ブランチ名の検証
```

**PRタイトル検証：** gitmoji + type の定義済み組み合わせ11種を配列として定義し、シェルの前方一致（文字列リテラル）で検証する。

**ブランチ名検証：** typeごとにパターンを分けて検証する。

| ブランチ種別 | パターン |
| --- | --- |
| `feature` / `fix` / `refactor` / `docs` / `chore` | `<type>/<issue番号>-<kebab-description>` |
| `hotfix` | `hotfix/<issue番号>-<description>` または `hotfix/<description>`（番号任意） |
| `release` | `release/v<major>.<minor>.<patch>` |
| `main` | 検証をスキップ |

**ワークフロー名：** `✅ CI - Pull Request`

## 理由

**検証対象をPRタイトルとブランチ名の2つに絞った理由：**
コミットメッセージ単体の検証（commitlint等）は、Squash mergeを採用している場合に実効性が低い。Squash merge後のコミットメッセージはPRタイトルから生成されるため、PRタイトルを検証する方が実質的な効果が高い。ファイルパスによる自動ラベリングはラベル体系の方針（ADR 0003参照）と合わないため対象外とした。

**PRタイトルを正規表現パターンではなく文字列リテラルの配列で検証した理由：**
gitmoji（絵文字）を正規表現のパターン文字列に埋め込むと、`⚡️` のような複数バイト絵文字がシェルの正規表現エンジンで正しく扱われない場合がある。配列に文字列リテラルとして格納し、`grep -P` でシェル変数展開後の文字列と前方一致で比較することで、絵文字のバイト表現に依存しない安全な検証ができる。

**`hotfix` のIssue番号を任意とした理由：**
本番障害の緊急対応では、Issueを起票してからブランチを切る時間的余裕がない場合がある。`hotfix` に限りIssue番号なしを許容することで、緊急時の運用負荷を下げた。

**`release` ブランチのパターンを `v` プレフィックスありに固定した理由：**
Semantic Versioningのタグ命名規則（`v{major}.{minor}.{patch}`）と統一し、リリースブランチ名からバージョンを直接読み取れるようにした。

**ワークフロー名を `✅ CI - Pull Request` にした理由：**
CIの「通過・失敗」を判定するゲートを表す絵文字として `✅` を選択した。`🚦`（信号機）も候補に挙がったが、GitHubのUIでは小さく見にくいため却下した。

## 却下した案

**commitlintによるコミットメッセージ検証**：Squash mergeを採用しているため、個々のコミットメッセージより最終的なPRタイトル（= Squash後のコミットメッセージ）の検証を優先した。

**正規表現パターンへの絵文字直接埋め込み**：`^(✨ feat|🐛 fix|...|⚡️ perf|...)` のように1つの正規表現にまとめる案。`⚡️` が複数バイト絵文字のため正規表現エンジンによって扱いが異なるリスクがある。文字列配列による前方一致の方が安全と判断し却下。

**`hotfix` にも番号を必須にする案**：緊急時の運用負荷が増すため却下。ただし可能な限りIssue起票してから対応することをワークフロー規約に明記している。

## 影響・注意点

- Rulesetsの `Require status checks to pass` を有効化することでCIチェックをマージのブロック条件にできる。現在は個人開発のため無効（将来的な有効化を検討）
- `⚡️ perf` の検証が正しく動作することは実際のCI実行で確認済み
- `chore` ブランチの検証ルール（`<type>/<issue番号>-<description>`）は `enhancement` Issueから派生した作業にも使用する（ADR 0002参照）
