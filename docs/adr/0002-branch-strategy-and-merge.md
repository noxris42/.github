# 0002. ブランチ戦略・マージ方法

日付：2025-04-14
状態：採用

---

## 背景

ブランチ運用の方針とPRのマージ方法を統一する必要があった。個人開発を基本としつつ、将来のチーム開発移行も考慮した設計が求められた。また `fix` と `hotfix` を分離するかどうか、ブランチtypeの種類をどう定義するかも検討対象だった。

## 決定

- ブランチ戦略として **GitHub Flow** を採用する（`main` ブランチのみ常設、`develop` は不使用）
- マージ方法は **Squash merge のみ** を採用する
- ブランチtypeは以下の7種類とし、`fix` と `hotfix` を分離する

| Type | 用途 |
| --- | --- |
| `feature` | 新機能の開発 |
| `fix` | バグ修正（計画的・Issue起票して対応） |
| `hotfix` | 本番障害の緊急修正（`main` から直接分岐） |
| `refactor` | リファクタリング（機能変更なし） |
| `docs` | ドキュメントのみの変更 |
| `chore` | 設定・依存関係・ツールの変更 |
| `release` | リリース準備（バージョン更新など） |

## 理由

**GitHub Flow採用**：`develop` ブランチを持つ Git Flow は個人開発・小規模チームには過剰。`main` を常にリリース可能な状態に保つ GitHub Flow がシンプルで管理しやすい。

**Squash merge採用**：ブランチ内の試行錯誤コミットを1つに集約でき、`main` の履歴を意味のある単位に保てる。

**`fix` / `hotfix` 分離**：緊急度と手続きの違いをブランチ名で明示するため分離した。どちらも Semantic Versioning の `patch` に対応する。通常のバグ修正は Issue 起票から始まる計画的な対応（`fix`）で、本番障害など即時対応が必要な場合のみ `hotfix` を使用する。

**`chore` / `refactor` の追加**：コミット規約の type と対応させることで統一感を出した。

## 却下した案

**Git Flow**：`develop` / `release` / `hotfix` ブランチを持つ複雑な運用。個人開発には過剰と判断した。

**Merge commit**：マージコミットが増えて履歴が煩雑になるため却下。

**Rebase merge**：コミットハッシュが変わり、Issue・PRとの紐付けが追跡しにくくなるため却下。

**`fix` / `hotfix` を統合**：緊急度の違いをブランチ名に表現できなくなり、運用上の判断が曖昧になるため分離を維持した。

## 影響・注意点

- Squash merge後のコミットメッセージはPRタイトルと同じフォーマット（`<gitmoji> <type>(<scope>): <subject>`）で記述する
- `hotfix` は `main` から直接分岐し、マージ後に削除する
- `bug` Issueから派生したブランチtypeは緊急度に応じて `fix` または `hotfix` を選択する
- `enhancement` Issueから派生したブランチtypeは作業内容に応じて選択する（`feature` / `chore` / `refactor` / `fix` など）
