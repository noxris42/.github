# 0013. 1つのIssueは1つのWorkを表す

| 項目 | 内容 |
| --- | --- |
| Status | Accepted |
| Date | 2026-07-14 |

## Context（背景）

Issueを作業の起点とする運用（ADR 0012参照）のもとで、1つのIssueがどの粒度の作業を表すかが論点になった。

「1 Issue = 1 Document（1つのIssueは1つの文書の変更に対応する）」のように厳密に定義しようとすると、実際の作業では不自然な分割が生じることが分かった。例えば、思想（Philosophy）を追加する際に、それに伴ってArchitectureの変更が必要になる場合、これを別々のIssueに分けると、本来一つのまとまった作業を不自然に分断することになる。

この問題に対し、「変更の主体となるドキュメント（Subject Document）」という考え方が導入された。ある変更において何が変更の主体であるかを明確にし、それに付随する変更は同一のIssue内で扱えるようにした。

初期検討では、この作業単位を「Task」と表現していたが、後続のWorkflow設計では、Issueを完了させるために必要な一連の作業を表す用語として「Work」に統一された。

---

## Decision（決定）

1つのIssueは、1つのWork（作業単位）を表す。

作業に複数の対象（例：複数の文書）が関わる場合でも、変更の主体（Subject）が単一であれば、付随する変更は同一のIssue内で扱ってよい。

Workの詳細な構造・ライフサイクル・他の成果物との一般的な関係は、本ADRの対象外とする。本ADRが定めるのは、IssueとWorkが1対1で対応するという粒度の定義のみである。

---

## Rationale（理由）

Issueの粒度を「変更対象の数」で厳密に定義すると（例：1つの文書につき1つのIssue）、本来ひとまとまりの作業が不自然に分割される。これは「Issueを起点に追跡可能な単位で作業する」というADR 0012の趣旨には合うが、作業の実態（何が主目的で、何がその付随的な帰結か）を反映しない分割は、かえって追跡性を損なう。

変更の主体（Subject）を基準にすることで、Issueの粒度は「対象の数」ではなく「作業の目的」で決まるようになり、実際の作業のまとまりと、追跡単位としてのIssueが一致する。

---

## Alternatives Considered（却下した案）

| 選択肢 | 採用しなかった理由 |
| --- | --- |
| 1 Issue = 1 Document（変更対象の文書ごとにIssueを分ける） | 主たる変更に付随する変更（例：Philosophy変更に伴うArchitectureの調整）まで別Issueに分割することになり、本来ひとまとまりの作業が不自然に分断される |

---

## Consequences（影響・注意点）

- Issueの粒度は、変更対象の数ではなく、変更の主体（Subject）が単一かどうかで判断する
- 複数の対象に変更が及ぶ場合、どれが変更の主体であるかをIssue作成時に明確にする
- Branchを使う場合のIssue・Branch・PRの対応関係は、本ADRの対象外とし、別途定義する
- Workという概念自体の包括的な定義（ライフサイクル、Branchを使わない場合の扱い、複数Issue・複数PRとの関係等）は、本ADRの対象外であり、引き続き将来の検討課題とする
