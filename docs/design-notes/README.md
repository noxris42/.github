# Design Notes（設計検討ノート）

本ディレクトリは、設計に関する議論・疑問・比較・保留事項を、任意の完成度・形式で保存する場所である。

---

## このディレクトリについて

- 検討の中で結論に至った内容が含まれていてもよい
- 内容が不完全・矛盾・陳腐化していることを許容する
- 文書構造・粒度は自由であり、作成自体も必須ではない
- `0000-template.md` は白紙から書き始めにくい場合の任意の記載例であり、design-notesの標準構造を定義するものではない

## ファイル命名規則

design-notesのファイル名は、識別と作成順の把握のため、以下の形式とする。

```text
<4桁連番>-<slug>.md
```

**Examples:**

```text
0001-adr-redesign-review.md
0002-database-comparison.md
```

- 番号は作成順の単純連番とし、4桁ゼロパディングを使用する
- 実際のノートは `0001` から開始する。`0000` はテンプレート専用の予約番号であり、実ノートの作成順連番には含めない（`docs/design-notes/0000-template.md`）
- 番号が表すのは「作成順」と「一意識別」のみとする
- 番号に重要度・分類・依存関係・確定度・ADRとの対応・Statusなどの意味を持たせない
- 欠番を埋めるために既存番号を再利用しない。既存ファイルの番号は並べ替えない
- `slug`部分は内容を識別できる名称とする
- 番号以外の文書構造・粒度は自由とする

## 正式な決定の根拠にはしない

design-notesの内容は、規範や正式な設計判断を成立させる根拠として依存しない。ADRの記録対象となる設計判断について、正式な判断履歴は `docs/adr/` を参照する。ただし、検討過程の補足資料として参照することは妨げない。ADRからdesign-notesを参照する場合でも、判断と理由はADR本文だけで理解できるようにする。

design-notesに含まれる設計判断がADRの記録対象となる場合は、ADRへ抽出する。ADRの記録対象となる基準は `docs/adr/README.md` を参照する。ADR対象外の内容は、必要に応じて責務を持つ現行Documentation（Philosophy・Architecture・Convention・Specification）へ直接反映してよい。

## Conventionへの準拠

design-notesはConvention（Markdown Conventionなど）への準拠を要求しない。
