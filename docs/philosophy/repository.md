# Repository Philosophy（リポジトリ設計思想）

## Purpose（目的）

このリポジトリは、単なるテンプレート置き場ではなく、GitHub運用における思想・方針・アーキテクチャ・規約・実装仕様・テンプレートを一貫した基準のもとで管理するための、共通運用基盤である。

個々のテンプレートや仕様を場当たり的に積み上げるのではなく、その背後にある「なぜそのように運用するのか」という思想を先に定義し、すべての設計判断がその思想に立ち戻れる状態を作ることを目的とする。

テンプレートよりも思想を優先する。
実装よりも設計を優先する。

この優先順位こそが、このリポジトリの存在意義そのものである。

---

## Vision（このリポジトリをどう見るか）

### Why GitHub（なぜGitHubを共通運用基盤とするか）

GitHubは、コードだけでなくドキュメント・履歴・議論・自動化を一つの場所で扱える。バージョン管理された変更履歴がそのままガバナンスの証跡になり、誰が・いつ・なぜ変更したかを追跡できる。

このリポジトリは、その特性を活かし、思想から実装までを単一のプラットフォーム上で一貫して管理するための場として、GitHubを選択している。

### Why Repository（なぜRepositoryを単位として管理するか）

Repositoryは、単なるソースコード置き場ではない。プロダクト・ドキュメント・ガバナンスを一つの単位としてまとめ、それぞれの責務を独立しつつも矛盾なく管理できる単位として扱う。

この捉え方は、本書だけでなく、今後設計する Repository Profile をはじめとする下位の仕様にも引き継がれる前提である。

---

## Scope（適用範囲）

Repository Philosophy は、このリポジトリ全体における最上位の文書であり、以降に作成されるすべての設計文書の前提となる。

適用対象は以下を含む。

- このリポジトリ自体の構造・運用方針
- `architecture` ・ `conventions` ・ `specifications` ・ `adr` として今後作成されるすべての文書
- 通常のリポジトリだけでなく、共通設定を管理する `.github` リポジトリ

ただし、本書が扱うのは「思想・判断基準・価値観」までであり、以下は本書の対象外とする。

- 具体的な実装方法
- 個別のルールや手順
- テンプレートやワークフローの仕様

これらは `architecture` ・ `conventions` ・ `specifications` といった下位文書に委ねる。本書は、それらの文書が拠って立つ「判断の根拠」を提供する位置づけにある。

このリポジトリの設計文書は、Philosophy → Architecture → Conventions → Specifications という階層を持ち、常に上位文書が下位文書の判断基準となる。下位文書の内容が上位文書と矛盾する場合は、下位文書側を見直す。

## Core Philosophy（基本理念）

### Long-Term Assets（長期資産）

本リポジトリで管理する情報は、一時的な成果物ではなく長期資産として扱う。

成果物は継続的な改善を前提とし、単発で消費されるのではなく、時間をかけて価値を積み上げていくものとして設計・運用する。

### Knowledge over Tacit Knowledge（暗黙知より明文化）

このリポジトリは、GitHub・Markdown・ADRといった仕組みを用いて、判断や知見を頭の中に留めず、文書として残すことを前提とする。

暗黙知に依存した運用は、属人化を招き、長期資産としての価値を損なう。なぜその判断に至ったかという経緯までを含めて明文化することで、知見は個人を越えてリポジトリ全体の資産になる。

### Concept First, Implementation Second（概念優先、実装は後）

実装の前に概念を設計する。アーキテクチャや規約、テンプレートの品質と一貫性は、その背後にある思想が定まっていなければ保証できない。

そのため、このリポジトリでは構造やルールを設計する前に、必ず「なぜそうするのか」を言語化し、十分な抽象化を行った上で、必要最小限の実装を進めることを原則とする。

土台が定まらないまま上物を作らない。これは作業順序の話ではなく、設計に対する基本姿勢である。

### Consistency over Preference（好みより一貫性）

個人の好みや、個別最適化された場当たり的なルールは、短期的には便利でも、長期的には保守性・可読性・再利用性を損なう。

すべての設計判断は、局所最適ではなくリポジトリ全体の一貫性を優先して行う。共通ルールと個別ルールが衝突する場合、まず共通ルールを疑うのではなく、共通ルールに合わせて個別ルールを設計し直すことを基本姿勢とする。

これは個人と集団の好みの調整ではなく、リポジトリ全体のガバナンスに関わる宣言である。

### Commonality with Explicit Exception（明示的例外を伴う共通化）

共通化は、品質と保守性を高めるための基本姿勢である。しかし、すべての状況を共通ルールだけで律することはできない。

このリポジトリでは、例外そのものを否定しない。ただし、例外は暗黙に許容するのではなく、常に明示された判断として扱う。何が例外であり、なぜ共通ルールから外れるのかを示せない変更は、例外ではなく単なる逸脱とみなす。

共通ルールと例外の具体的な扱い方は、下位文書である Conventions や Specifications に委ねる。本書が定めるのは、その前提となる「共通化を優先しつつ、例外を明示することで初めて許容する」という考え方である。

### Simplicity over Complexity（複雑さより単純さ）

将来の拡張性を考慮しつつも、現時点で必要以上に複雑な仕組みを導入しない。

複雑さは将来の柔軟性を生むように見えて、実際には理解と保守のコストを増大させることが多い。必要になった時点で必要な分だけ拡張できる単純さを優先する。

### Single Source of Truth（単一情報源）

同じ情報を複数箇所で管理しない。情報は、それぞれ責務を持つ唯一の場所で一元的に管理する。

情報の重複は、更新漏れや矛盾を生み、長期資産としての信頼性を損なう最大の要因となる。

### Design for Machine Processing（機械的処理による運用を前提とした設計）

このリポジトリは、人間だけでなく、自動化ツールやAIなど、機械的処理による利用も前提として設計する。philosophy・architecture・conventions・specifications・adr という明確な責務分離は、人間にとっての可読性だけでなく、機械的な処理主体が文脈を誤らずに参照・実行できることを意図したものである。

曖昧さを許容しない構造化は、人間と機械的処理主体の双方にとっての信頼性の基盤となる。

### Evolve Through Operation（運用を通じた進化）

設計・規約・ドキュメントは完成品ではない。実際の運用を通じて得られた知見をもとに改善し、継続的に成熟させていくものとして扱う。

これは Concept First, Implementation Second と矛盾しない。先に概念を固めるのは、変更を否定するためではなく、変更が場当たり的に行われることを防ぎ、改善を一貫した方向へ積み重ねるためである。

本書は、リポジトリ全体のすべての設計判断がそこに立ち戻れる、出発点である。
