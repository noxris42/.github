# Documentation Philosophy（ドキュメント設計思想）

## Purpose（目的）

このリポジトリで管理するドキュメントは、一時的な作業メモや手続き上の成果物ではない。判断・知見・設計の背景を共有・継承可能な形で残し、リポジトリ全体の知識基盤を構成する長期資産である。

本書は、そのようなドキュメントをどのような価値観・設計思想のもとで設計・管理するかを定義する。「良いドキュメントの書き方」ではなく、「良いドキュメントとは何か」という判断基準を提供することを目的とする。

本書は Repository Philosophy を前提とし、そこで定義された思想をドキュメントという資産へ適用する。Markdown記法・テンプレート・ドキュメント構造の具体的な定義は、それぞれ Conventions・Specifications・Architecture に委ねる。

---

## Scope（適用範囲）

本書は、このリポジトリで管理するすべてのドキュメントに適用される。

対象は以下を含む。現時点で列挙されていないドキュメントであっても、このリポジトリで管理される文書である限り、本書の思想が適用される。

- README
- CONTRIBUTING
- Philosophy・Architecture・Conventions・Specifications ドキュメント
- ADR
- その他本リポジトリで管理するすべての文書

本書が扱うのは「ドキュメントをどう考えるか」までであり、以下は本書の対象外とする。

- ドキュメント体系・構造・責務の定義 → Architecture
- Markdown記法・文書構成・命名規則 → Conventions
- テンプレート・メタデータ・自動生成仕様 → Specifications

---

## Core Philosophy（基本理念）

### Documentation as Long-Term Asset（ドキュメントの長期資産化）

ドキュメントは、書かれた時点で完成する成果物ではなく、継続的に保守・改善される知識資産として扱う。

一時的な正確さよりも、長期にわたって参照・更新され続けられる持続可能な設計を優先する。ドキュメントの価値は、書かれた瞬間ではなく、将来にわたって活用され続けることで生まれる。

### Knowledge over Tacit Knowledge（暗黙知より明文化）

Repository Philosophy で定義した「Knowledge over Tacit Knowledge」の思想を、ドキュメント設計へ適用する。

ドキュメントの本質的な役割は、暗黙知を共有・継承可能な知識資産へ変換することである。結論だけを残すドキュメントは不完全であり、なぜその結論に至ったかという判断の背景・前提・根拠を含めて初めて、知識は個人を越えてリポジトリ全体の資産となる。

### Reader-Oriented Communication（読者指向コミュニケーション）

ドキュメントは書き手のためではなく、読み手のために設計する。

読み手とは、未来の自分・共同開発者・AIなど、知識を共有する相手すべてを指す。書き手が知っていることを書くのではなく、読み手が理解できることを書く。ドキュメントはその読み手との、時間と場所を越えたコミュニケーション手段である。

### Context over Content（内容より文脈）

情報量よりも文脈を優先する。

個々の情報を列挙するだけでは、読み手はその情報を正しく判断に活かせない。「なぜその情報が存在するのか」「どのような前提のもとで成立するのか」という文脈こそが、読み手が同じ判断に到達するために必要なものである。ドキュメントは結論の置き場所ではなく、判断を支援するための構造物として設計する。

### Separation of Concerns（責務の分離）

一つのドキュメントは一つの主要な責務を持つ。責務の重複は保守性を低下させ、Single Source of Truth を損なう。

同じ情報を複数の文書に重複して記載しない。情報は責務を持つ唯一の場所で管理する。どのドキュメントが何を責務として持つかを明確に分離することが、ドキュメント体系全体の一貫性と保守性を支える。

### Information Design First（情報設計優先）

Markdownは情報の表現手段であり、設計思想ではない。

ドキュメントを設計する際は、まず「何を伝えるか」「どのような構造で情報を整理するか」を先に決める。情報設計が先にあることで、形式に引きずられた中身のない構造を防ぐ。

### Coexistence with Machine Processing（機械的処理主体との共存）

ドキュメントは、人間だけでなく、AIや自動化ツールなど機械的処理による利用も前提として設計する。

曖昧な構造・暗黙の文脈・属人的な表現は、機械的処理主体にとっての障壁になるだけでなく、人間にとっての可読性をも損なう。意味が明確に伝わる構造を優先することは、あらゆる読み手に対する誠実さである。

### Evolve Through Use（使用を通じた進化）

ドキュメントは完成品ではない。実際に読まれ・参照され・更新される運用を通じて継続的に改善し、成熟させていくものとして扱う。

初回の完成度に固執するより、運用の中で得られた知見をドキュメントへフィードバックすることを優先する。ドキュメントの品質は、書かれた時点ではなく、使われ続ける中で高まる。

本書は、このリポジトリの知識基盤を支えるすべてのドキュメント設計判断の、出発点である。
